# v0.1.1 Requirements

These requirements are based on operational feedback from a real host deployment of v0.1.0. The goal for v0.1.1 is not to broaden scope, but to remove the most immediate friction points that block reliable deployment and operation.

## 1. Linux Release Publish Must Be Independently Releasable

Apple Darwin should not block publication of `x86_64-unknown-linux-gnu`. The enclave host only needed the Linux artifact, and waiting on a full release matrix delayed a real deployment for no operational benefit.

Requirement:

- Linux artifacts must be publishable independently of Darwin artifacts.
- A failing or delayed Darwin build must not prevent Linux release publication.

## 2. NixOS Compatibility Should Be First-Class

v0.1.0 currently runs on NixOS only via explicit `nix-ld` loader invocation:

```bash
/lib64/ld-linux-x86-64.so.2 /var/lib/theorem/current/theorem-node ...
```

That is workable, but it is not yet a clean supported path.

Requirement:

- v0.1.1 should provide at least one explicitly supported NixOS story.
- Acceptable paths include:
  - a statically linked Linux binary
  - a Nix package or flake output
  - documentation that explicitly blesses the `nix-ld` launch contract

## 3. Health Endpoint Semantics Should Be Explicit

`/health` currently returns `404` while the server is healthy. That is operationally awkward because it creates ambiguity for monitoring, orchestration, and host-level checks.

Requirement:

- If `/health` is intended to exist, it should return `200` with a minimal status body.
- If `/health` is not intended to exist, the canonical liveness/readiness command must be documented instead.

## 4. Release Contract Should Be Stable and Documented

Deployment automation depends on predictable release outputs. Right now, the release contract is not explicit enough for safe automation.

Requirement:

- Document the release contract for:
  - asset naming
  - checksum file naming
  - SBOM presence
  - supported targets
  - whether `latest` is safe to automate against

## 5. Release Admission Needs to Become Real Quickly

`bootstrap`, `status`, and `serve` are enough to stand up the runtime, but the next product step is moving from "runtime exists" to "runtime is authoritative."

Requirement:

- Prioritize the next operationally meaningful admission primitives:
  - artifact hash verification and admission
  - capability release
  - canonical commitment or state export
  - narrow irreversible action admission

## 6. Secret Handling Needs a Stronger Long-Term Story

Requiring `THEOREM_COMMITMENT_SECRET` is acceptable for now, but the long-term operating model is still unclear.

Requirement:

- Define the intended secret handling model, including whether the long-term path is:
  - environment variable only
  - file input
  - KMS or sops integration
  - sealed or bootstrap-derived secret material

## 7. Operational Docs Should Distinguish Quickstart From Production

The operational path for a production enclave host is meaningfully different from a developer quickstart. Those two modes should not be conflated.

Requirement:

- Provide production-oriented operational documentation that covers:
  - release-only install
  - checksum verification
  - versioned install directory layout
  - stable symlink conventions
  - `systemd` unit examples
  - upgrade and rollback guidance

## Summary

The priority for v0.1.1 is to make Linux deployment operationally reliable, make NixOS support explicit, remove health-check ambiguity, stabilize the release contract, and improve the production deployment story. The release does not need a broad feature expansion, but it does need to reduce the friction that currently makes real host deployment harder than it should be.
