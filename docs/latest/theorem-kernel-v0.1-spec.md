# Theorem Kernel v0.1

## Primitives

**Delegation** — a bounded representation of what a principal has authorized an agent to do. Opaque to the kernel. The kernel does not interpret it. Immutable once accepted. To change what an agent may do, revoke and re-delegate. Each delegation is assigned a kernel-generated identifier (delegation ID) at acceptance time. Delegation IDs are unique within the kernel instance and are used for chain tracing, revocation references, and the chain fingerprint in Expiry deduplication.

**Action** — a request by an agent to do something with effect outside itself.

**Agent** — any entity that receives a delegation and requests actions. The kernel does not distinguish between agent types. Every agent is bound to an identity. The identity binding is presented to the kernel at Introduce time and verified structurally at every subsequent operation: the kernel confirms the presented binding bytes match the agent's registered binding bytes exactly. "Structural verification" means byte-equality — the kernel does not interpret the binding's internal format or validate its cryptographic authenticity. The authenticity of the identity binding (i.e., that the binding genuinely represents the agent's identity) is an axiomatic trust dependency with the same status as the comparison function. An operation from an unbound or mismatched identity is rejected.

**Principal** — an Agent at the root of a Chain. It issues the first Delegation. It does not receive a Delegation from any other agent. The Principal's identity is an axiomatic input at Genesis. The Principal holds the genesis delegation by fiat — this is a declared bootstrap asymmetry, not a delegation operation. The genesis delegation has no issuing agent and no issuer-based revocation path.

**Chain** — the path from the root principal to the acting agent through the delegation tree. Traced at Check time. Ordered, acyclic — no agent appears more than once in any chain. Every link was mediated by the kernel.

**Attestation** — a tamper-evident record of a kernel operation within the stated integrity boundary. Written atomically with the operation it records. Attestation is the integrity mechanism for the entire kernel. If Attestation fails, no kernel guarantee holds. The attestation log is a total order of all kernel operations. Single node execution and serialized operations guarantee this ordering. The kernel does not self-verify attestation integrity — it depends on the substrate for tamper-evidence (see Substrate Requirements: Integrity). If the substrate detects attestation tampering, the kernel halts. No further operations are accepted. Recovery requires a new kernel instance with the tampered attestation log as auditable input.

**Genesis** — the kernel's starting condition. Accepts five immutable inputs: the Principal's identity binding, the genesis delegation, the Expiry interval (in the same unit as the substrate's monotonic clock), the comparison function, and the comparison timeout (in the same unit as the substrate's monotonic clock). Attested as root. Exactly one per kernel instance. This is the only Attestation without a preceding operation. Multiple trust domains require multiple kernel instances. Genesis cannot be revoked — there is no issuing agent. A Return against the genesis delegation is terminal: the action will always Return for this kernel instance because the genesis delegation is immutable and the comparison function is deterministic — the same inputs will always produce the same outcome. This is a logical consequence of immutability, not a separate enforcement mechanism. The kernel does not maintain a blacklist; it simply re-evaluates and re-Returns on each submission (subject to Expiry deduplication). The agent is not blocked from other actions within the delegation. To change the system's boundary, start a new kernel instance.

**Expiry** — a Return that receives no response from the principal does not wait indefinitely. The kernel terminates unresolved Returns after the Expiry interval declared at Genesis. The interval is immutable within the kernel instance. The kernel attests the termination. After Expiry, an identical re-submission is rejected within one Expiry interval from the moment of termination. Identity is defined as: same agent identity, same action content (byte-equal), same delegation, and same chain fingerprint. The chain fingerprint is a hash of the ordered delegation IDs from root to the acting agent's delegation. This ensures that a chain change (upstream revoke + re-delegate) produces a different fingerprint even when the agent's own delegation is byte-identical, allowing legitimate re-submission under a new chain. The kernel hashes this quadruple at Expiry and rejects matches. The rejection is attested. Consequence: when an agent receives a new delegation (via re-introduction and re-delegation), the chain fingerprint changes even if the delegation content is identical. The new chain produces a different hash, so deduplication does not block re-submission under the new authorization context. This is correct — a new chain is a new authorization decision. Expiry deduplication hashes are pruned after the rejection window elapses — the kernel deletes a hash once the monotonic clock has advanced one full Expiry interval past the Expiry termination time. Correctness of deduplication depends on delegation ID uniqueness (see Delegation primitive): because each Delegate operation produces a unique delegation ID, a revoke-and-re-delegate always changes the chain fingerprint, and a previously blocked hash cannot recur under a structurally different chain.

**Pending Notification Queue** — declared kernel state that holds Return notifications awaiting delivery to the principal. Each entry records: the acting agent's identity, the action content, the failing chain link, and the attestation entry ID of the originating Check. The queue is bounded by unique (agent, action content, delegation) triples — duplicate Returns for the same triple are coalesced rather than appended. Enqueue and delivery are both attested. The queue exists because the kernel is synchronous: a Return for a non-principal agent's Check cannot be delivered to the principal in the same transaction. This is the declared mechanism that fulfills the Kernel Interface's delivery promise.

## Operations

**Introduce** — register an agent with the kernel and bind its identity. Accepts the agent's identity binding and a reference to the introducing agent (who must have an intact chain). The kernel records the agent's identity and makes it eligible to receive a delegation. Attested atomically. The principal is introduced at Genesis and does not require this operation. An agent that has not been introduced cannot receive a delegation. An introduced agent that has never received a delegation can request no operations other than receiving a delegation. Introduce is idempotent for an already-registered agent whose delegation has been broken: the kernel recognizes the agent's existing identity binding, verifies it matches the presented binding, and resets the agent's delegation slot to empty — making it eligible to receive a new delegation from a different chain. The introducing agent need not be the same as the original introducer. If the presented identity binding does not match the registered binding, the operation is rejected. This re-introduction path is what enables the "new Introduce-Delegate sequence" referenced by the Delegate operation.

**Check** — compare the action against every delegation in the chain using the comparison function. The comparison function is called once per link in the chain, receiving the action and a single delegation, and returning Pass or Return. The action Passes only if every invocation returns Pass — if any returns Return, the overall result is Return. Produces one of two outcomes:
- **Pass** — the action is within every delegation in the chain. The kernel attests the Pass. Execution of the action is outside the kernel's scope. The kernel does not attest execution, only authorization. The kernel sends the Pass result to the requesting agent through the kernel interface.
- **Return** — the action is outside at least one delegation in the chain. The kernel sends a Return notification to the root principal through the kernel interface. The notification includes: the acting agent's identity, the action content, and the link that failed. No intermediate agent sees or resolves the Return. The kernel attests the notification. What the principal does with a Return is outside the kernel's scope. The principal may subsequently issue Delegate, Revoke, or other kernel operations — these are independent kernel operations, not linked to the Return.

If the principal's action against the genesis delegation produces Return, the genesis-terminal rule applies: the action is permanently denied. The no-self-resolution rule does not independently govern this case — it is subsumed by the genesis-terminal mechanism.

For non-principal Returns: the Return is informational. There is no suspended condition. The agent whose action was Returned may re-submit the action at any time (subject to Expiry deduplication if a previous identical submission expired). The kernel does not track Return state or wait for resolution. Each Check is a stateless evaluation of the chain at invocation time. If the chain has changed since the last Check (due to Delegate, Revoke, or other operations), a re-submitted action may produce a different result.

Any outcome that is not clearly within the delegation is Return. The comparison function is external to the kernel. The delegation is opaque to the kernel, not to the comparison function. The comparison function must be deterministic: same action and same delegation always produce the same outcome. The comparison function must terminate within the comparison timeout declared at Genesis. Timeout produces Return. The attestation distinguishes timeout-Return from policy-Return: the attested outcome records whether the Return was caused by the comparison function returning Return or by timeout. The notification to the principal does not distinguish — both are Return with the failing link. The kernel's guarantees depend on the comparison function being correct and the identity binding being valid. These are co-dependencies — both are required at Check time and cannot be resolved independently. A dishonest comparison function or a forged identity defeats the kernel. These are stated dependencies, not guarantees. The comparison function has the same trust status as the kernel itself — it can silently Pass any action regardless of delegation content. This is accepted as a declared axiom, not a hidden assumption.

**Delegate** — an agent issues a delegation to a downstream agent. The receiving agent must have been Introduced. The kernel verifies the issuing agent's chain is intact before accepting. Attested atomically. The kernel records the delegation without checking it against the parent delegation's content. Validity of content is determined at Check time when the downstream agent acts — this means an agent can issue a delegation broader than its own authority. The downstream agent's actions will simply Return at the narrower upstream link during Check. The kernel provides no feedback about this at Delegate time; this is a consequence of delegation opacity. The delegation structure is a tree, not a DAG. An agent may hold at most one active delegation. An agent may issue multiple delegations to different downstream agents (this is how the tree branches). The kernel rejects a delegation if the receiving agent already holds any delegation (active or inactive), already appears in the issuing agent's chain, or if the issuing agent's chain has a gap. An agent with an inactive delegation must be re-introduced first to clear the delegation slot — this prevents dual-delegation states that would violate the tree invariant if the original chain were later restored (subtree resurrection). This is why the Delegate operation references "a new Introduce-Delegate sequence."

**Revoke** — the issuing agent rescinds a delegation it issued. Only the agent that issued a delegation may revoke it. The kernel verifies the issuing agent's chain is intact before accepting. Attested atomically. Downstream delegations are not automatically invalidated — they become inactive (their chains have a gap) and fail Chain integrity on next Check. The kernel does not notify downstream agents. When a revocation orphans a subtree, no agent gains automatic authority over the orphaned delegations. Each orphaned delegation can only be revoked by its own issuing agent (whose chain must be intact to do so). If an orphaned agent's issuing agent is itself orphaned, the delegation persists in inactive state until either: (a) the chain is restored by re-delegation upstream, allowing the issuing agent to revoke, or (b) the orphaned delegation is abandoned — it remains in the kernel's records as inactive but produces no effects.

Subtree resurrection: if a revoked agent receives a new delegation (via re-introduction and re-delegation), downstream delegations whose chains pass through that agent become traceable again — their chains are restored. This is not an active reactivation: the kernel does not scan for or modify downstream delegation records. Resurrection is a passive consequence of chain traceability. The chain is traced by following each agent's current delegation slot to its issuer; when a gap in this pointer chain is filled by a new delegation, all agents reachable through the restored path have intact chains again. A delegation record orphaned by re-introduction (where the receiving agent's delegation slot was cleared and reassigned to a different delegation) is not resurrected — it is unreferenced and produces no effects, even if the original issuing agent later regains an intact chain. Only delegations whose holders still reference them through their delegation slot participate in chain tracing. This is intended behavior. Revocation is local to the revoked link, not transitive. The comparison function provides the safety guarantee: at Check time, every link in the restored chain is evaluated, so an action that exceeds the new (possibly narrower or broader) delegation is still Returned at the constraining link. If the restored chain is broader than the original at every link, downstream actions that previously Returned may now Pass — this is correct, because every authorizer in the new chain has approved. Administrative awareness of resurrection is a derived layer concern. Resurrection is observable through the attestation log: the Delegate operation that fills the gap is attested, and subsequent Check operations against the restored chain record the full chain state.

**Attest** — written atomically with every kernel operation. Records: the operation performed, the inputs to that operation, the outcome, and the chain state at that moment. Chain state is operation-specific: for Check, the acting agent's full chain; for Delegate, the issuing agent's chain (pre-delegation); for Revoke, the issuing agent's chain and the revoked delegation ID (post-revocation state); for Introduce, the introducing agent's chain (the new agent has no chain yet); for Genesis, no chain (this is the root attestation).

## Kernel Interface

The kernel exposes a synchronous request-response interface. All operations are submitted through this interface and results are returned through it. The interface is the kernel's boundary with its callers.

**Submission**: An agent submits an operation request (Check, Delegate, Revoke, Introduce) with its identity binding and the operation's inputs. The kernel authenticates the agent's identity, validates the request, executes the operation, attests it, and returns the result — all within a single serialized transaction.

**Results**: Every operation returns a result to the submitting agent: Pass or Return for Check, acceptance or rejection for Delegate/Revoke/Introduce. Return notifications to the principal are sent as the result of the Check operation — the principal receives the notification when the kernel processes the Check, not as a separate asynchronous delivery. If the submitting agent is not the principal, the Check result returned to the submitting agent is simply "Return" (without the notification details sent to the principal).

**Pending notification queue**: When a non-principal agent's Check produces a Return, the kernel queues a notification for the principal. This queue is declared kernel state — it is attested when notifications are enqueued and when they are delivered. The queue is bounded by uniqueness: it holds at most one notification per unique (agent, action content, delegation) triple; duplicate Returns for the same triple are coalesced rather than appended. Coalescing bounds the queue per unique triple, but the action content dimension is unbounded — an agent can submit arbitrarily many distinct actions, each producing a unique queue entry. The effective bound is the rate of unique Check-Returns between principal deliveries, not a fixed structural limit. Delivery drains the queue; Expiry independently terminates unresolved entries. The principal's pending notifications are delivered the next time the principal submits any operation — they are included in the response and the delivery is attested as a sub-step of the principal's operation (within the same atomic transaction). After delivery, the queue is cleared. The Expiry mechanism independently terminates unresolved Returns regardless of whether the principal has received the notification — notification delivery is best-effort within the kernel's synchronous model.

**Concurrency**: The kernel accepts one operation at a time. Concurrent submissions are queued and processed in arrival order. The queuing mechanism is a substrate concern.

## Rules

**Atomicity** — every kernel operation and its Attest are one indivisible unit. Kernel operations are serialized. One operation completes before the next begins.

**Chain integrity** — every delegation traces to a root principal. No gaps. A gap exists when any delegation in the chain has been revoked, or when the issuing agent of any delegation in the chain holds no active delegation themselves.

**No self-resolution** — an agent cannot resolve a Return signal about its own delegation. In the current model, this rule is subsumed by two other mechanisms: (a) Returns for principal actions against the genesis delegation are terminal by the genesis-terminal rule, and (b) Returns for non-principal actions are informational notifications to the principal, who is resolving another agent's delegation, not its own. The rule is retained as a named invariant for future-proofing: any extension to the model that introduces new resolution mechanisms must preserve this property.

Delegation precision determines system usability. The kernel does not enforce precision. The system creates the incentive.

The kernel guarantees integrity, not liveness. If the principal is unresponsive, the system halts. Liveness detection and recovery are derived layer concerns.

## Substrate Requirements

The kernel operates on a single authoritative node. Distribution, replication, and redundancy are derived layer concerns outside the kernel's scope.

The kernel requires four properties from whatever it sits on. How they are satisfied beyond the stated boundaries is outside the kernel's scope.

**Atomicity** — every kernel operation either completes fully or does not happen. No partial states. Guaranteed by single node execution.

**Persistence** — attestation records survive software failure. The substrate must provide fsync semantics. The storage device must provide power-loss safe writes. Hardware failure beyond this boundary is a derived layer concern handled by replication above the kernel.

**Integrity** — delegations and attestation records cannot be modified without detection by any actor without privileged substrate access. The kernel's attestation primitive provides the detection mechanism. The substrate must ensure the attestation record itself cannot be silently modified. If modification is detected, the substrate must signal the kernel, which halts (see Attestation primitive). Integrity against privileged substrate access requires physical access control. That is outside the kernel's scope and outside its guarantees.

**Time** — the substrate must provide a monotonic clock. Expiry and comparison timeout depend on it. The Expiry interval and comparison timeout declared at Genesis are in the same unit as this clock. The accuracy and synchronization of the time source are substrate concerns. The kernel requires only that time moves forward. The relationship between clock units and wall-clock time is a deployment concern outside the kernel's scope.

**Concurrency control** — the substrate must provide a mechanism to serialize concurrent operation submissions (e.g., a mutex or queue). The kernel processes one operation at a time; the substrate ensures this invariant at the boundary.
