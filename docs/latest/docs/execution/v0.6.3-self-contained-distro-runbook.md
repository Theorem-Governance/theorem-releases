# `v0.6.3` Self-Contained Distro Runtime Runbook

## Purpose

This runbook defines the complete closure path for `v0.6.3` as one runtime
system, not a staged checklist.

The release is real only if the exact shipped artifact family boots on Hetzner,
constructs its governed runtime from shipped metadata, admits the live gateway
session path used by `validation-probe`, and emits the governed proof bundle
from inside that running image.

This runbook is runtime-dependent by design:

- every step exists because a later runtime path consumes it
- every prerequisite is tied to a concrete command, config file, or admitted
  payload
- local validation must execute the governed proof path, not just inspect
  archive contents
- Hetzner validation is a confirmation step for the exact shipped image, not a
  discovery loop

## Runtime Spine

The release depends on this exact chain:

1. one exact artifact family is built from one source state
2. the staged rootfs carries the exact runtime binaries, store objects, rc.d
   scripts, env files, manifests, and bootstrap metadata consumed on first boot
3. first boot runs `theorem_runtime_prepare`, which generates runtime env,
   bootstraps theorem-node if needed, and seeds live governed service runtime
4. theorem-init launches theorem-node, then theorem-gateway, using the shipped
   service hashes and runtime authority metadata
5. `theorem-govagent validation-probe` calls `POST /ring3/intent`
6. theorem-gateway creates the bus session principal for that request
7. theorem-bus executes `register_session`
8. theorem-node admits `bus.session.register` for that exact runtime payload
9. the intent/outcome path completes and emits accepted and rejected AI-act
   evidence
10. `theorem-init governance collect-release-proof` exports the governed proof
    bundle from the live host
11. release evidence is copied back and validated against the exact shipped
    assets

If any link in that chain is implicit, guessed, or tested only in source, the
runbook is incomplete.

## Target Outcome

`v0.6.3` is complete only when all of the following are true at once:

- installer tarball, upgrade bundle, single-binary tarball, and direct-write
  image all refer to one exact source state and one exact runtime hash set
- the shipped direct-write image is self-contained by default
- `first-boot.conf` is override-only, not required for deployability
- the shipped image reaches a healthy governed runtime on Hetzner
- the shipped runtime launches `theorem-gateway` through live governed service
  admission
- the shipped govagent binary exposes and successfully runs
  `validation-probe`
- the governed proof path produces both rejected and accepted AI-act evidence
  from inside the enclave
- the governed proof bundle and hardware validation report are bound to the
  exact final direct-write image hash
- `scripts/validate-release.sh` passes for the exact release directory with the
  collected evidence attached

## Runtime Rules

- Do not do another Hetzner write until the exact shipped artifact family has
  passed the full governed proof path locally.
- Do not treat archive presence checks as runtime validation.
- Do not validate a near-match image. Use the exact final image hash only.
- Do not collect proof from source-only functionality. The proof must come from
  the shipped image and shipped binaries.
- Do not leave the gateway session principal implicit. The seeded runtime
  authority must match the principal and payload the live gateway actually
  creates.
- Do not assume gateway service-runtime admission is sufficient for the proof
  path. The session admission used by `validation-probe` must be seeded
  explicitly.
- Do not assume local model transport uses TLS. The shipped
  `theorem://validation-probe` path requires `GOVAGENT_REQUIRE_TLS=false`.
- Proof collectors must export the same runtime env used by the shipped
  govagent path, including `GOVAGENT_REQUIRE_TLS`.
- Commit release-critical source and automation changes before another long
  rebuild/deploy loop.

- all `.sha256` files

The staged rootfs, theorem-store object manifests, activation manifest, and

lanes.

### 2. First-Boot Runtime Preparation

The staged rootfs must contain:

- `theorem-node`
- `theorem-gateway`
- `theorem-govagent`
- all required rc.d scripts
- helper scripts used by runtime prepare and auto-bootstrap
- theorem-store objects for governed launch artifacts
- runtime metadata written by `make-release.sh`

`theorem_runtime_prepare` must be able to:

- generate `/etc/theorem/theorem-node.env`
- generate `/etc/theorem/theorem-gateway.env`
- generate `/etc/theorem/theorem-govagent.env`
- run bundled bootstrap when the DB is empty
- seed the runtime authority required for live governed launch

### 3. Exact Gateway Session Admission

The governed proof path depends on the live ingress sequence, not just on
service startup:

`validation-probe`
-> gateway `POST /ring3/intent`
-> gateway session principal creation
-> bus `register_session`
-> kernel admission for `bus.session.register`

The runbook must treat that as one release-critical dependency chain.

Required properties:

- the exact session principal created by theorem-gateway is known
- the runtime seeding admits that exact principal and payload, or the admission
  pattern is widened intentionally and narrowly
- the admission shape is deterministic and derived from the runtime path, not
  operator memory
- this path is exercised before Hetzner from the shipped artifact set

### 4. Govagent Runtime Environment

The shipped proof path must expose the runtime env required by
`validation-probe`, including:

- `GOVAGENT_ENTITY_ID`
- `GOVAGENT_MODEL_URL`
- `GOVAGENT_MODEL_NAME`
- `GOVAGENT_TASK_PORT`
- `GOVAGENT_CORPUS_PATH`
- `GOVAGENT_GATEWAY_URL`
- `GOVAGENT_GATEWAY_TOKEN`
- `GOVAGENT_SEED_PATH`
- `GOVAGENT_WITNESS_TOKEN`
- `GOVAGENT_TASK_TOKEN`
- `GOVAGENT_KERNEL_URL`
- `GOVAGENT_REQUIRE_TLS=false`

If the proof collector or operator command path omits `GOVAGENT_REQUIRE_TLS`,
the local-model proof path is not validated.

### 5. Proof Collector Behavior

The proof collector must match the shipped runtime assumptions:

- rescue and installed-system SSH host keys must be handled explicitly
- governed proof export must come from `theorem-init governance collect-release-proof`
- AI-act evidence must come from the shipped
  `theorem-govagent validation-probe`
- local validation must fail if the shipped artifact set cannot execute the
  governed proof path end to end

This procedure must complete before the final Hetzner write.

### 1. Close Source And Automation Gaps

Verify and fix, if needed:

- `theorem-govagent` CLI wiring exposes `validation-probe`

- focused tests for theorem-node runtime/service admission
- focused tests for theorem-gateway live ingress session setup if changed

Rebuild together:

- single-binary theorem-node tarball

### 4. Perform A Full Local Governed Proof Dry Run

This is mandatory before Hetzner.

Run the proof path from the shipped artifact set, not from source-tree
binaries:

- boot or stage the shipped runtime in the same way the final image will use it
- start the governed runtime from shipped configs and manifests
- execute `theorem-init governance collect-release-proof`
- execute `theorem-govagent validation-probe`
- confirm the live ingress path completes through gateway session setup and
  bus session registration
- confirm both rejected and accepted AI-act files are emitted
- confirm `service-admissions.json` contains live governed admissions relevant
  to the runtime

The local dry run is not complete if it stops at artifact extraction,
`--help`, or archive parity.

## Hetzner Validation Procedure

Hetzner exists to validate the exact shipped image on real hardware, not to
discover missing runtime dependencies.

### 1. Write The Exact Final Image

Required environment:

- Hetzner server id `125042903`
- IPv4 `5.161.121.1`
- rescue path via `hcloud`
- fresh rescue and live `known_hosts` files

Actions:

- enable rescue on the target server
- power cycle into rescue
- upload the exact final direct-write image and `.sha256`
- upload `scripts/write-direct-write-image.sh`
- upload optional `.tmp-v0.6.3-first-boot.lf.conf`
- run the rescue writer against `/dev/sda`
- disable rescue
- reboot into the installed system
- reconnect using a fresh installed-system host-key file

### 2. Validate First-Boot Runtime

Run:

- `hostname`
- `cat /etc/theorem/release.version`
- `zpool list theorem`
- `ps auxww | egrep 'theorem-(init|node|gateway|govagent)'`
- `sockstat -4 -l | egrep '3170|3180|3181'`
- `fetch -qo - http://127.0.0.1:3170/health`
- `tail -n 120 /var/log/theorem/theorem-init.log`

Success conditions:

- root pool expanded to full disk
- theorem-node healthy on `3170`
- theorem-gateway listening on `3180`
- theorem-init log records live theorem-node authority cutover
- theorem-init log records admitted service launch for `theorem-gateway`
- the authority source for the gateway launch is
  `live-governance-act`

### 3. Collect Governed Proof On Box

Run on the target box:

## Known Failure Modes To Defend Against

- source contains a feature that the shipped binary does not
- one artifact lane was rebuilt while another kept stale runtime binaries
- theorem-store object hash does not match shipped runtime binary hash
- runtime admission path uses a hash encoding that differs from the one queried
  by theorem-init
- gateway service launch is admitted, but the proof path session-admission
  payload is not
- the live gateway session principal differs from the seeded runtime authority
- govagent proof command exists in source but not in shipped binary
- proof collector fails due to rescue or live host-key churn
- `GOVAGENT_REQUIRE_TLS=false` is missing from the shipped or exported proof
  environment
- release metadata still contains stale "pending validation" language after
  proof is complete
- a long external wait is mistaken for progress
